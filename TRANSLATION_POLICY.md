# Translation Policy (Agent Guide v2.2)

## 0) 목적
- 목표는 “번역문”이 아니라 **한국어로 처음부터 쓰인 글**처럼 읽히는 결과입니다.
- 의미(사실/제약/의도)를 보존하되, 문장 구조·리듬·수사는 한국어 관습에 맞게 재작성합니다.
- 문체는 고정값이 아니라 **문서 스타일 시트(Style Sheet)의 선택 결과**입니다.
- 품질 하한을 보장하기 위해 **하드 게이트(통과/탈락)**를 둡니다.

---

## 1) 기본 방향
- Source: English
- Target: Korean

---

## 2) 우선순위 (Strict)
1) 자연스러운 한국어 가독성/리듬 (번역투 제거)
2) 의미 충실도 (의도/제약/사실 보존)
3) 용어 일관성 (글로서리 우선)
4) 수사/톤 재현 (한국어식 등가치환)

### 2.1 충돌 해결
- 가독성과 의미가 충돌하면: **의미를 보존한 채 문장을 바꿉니다.**
- 용어가 어색하면: **문장을 바꿔 용어를 수용**합니다. (용어 억지 변경 금지)
- 수사가 직역되어 어색하면: **효과(설득/경고/회고)를 기준으로 한국어식으로 등가치환**합니다.

---

## 3) 작업 절차 (필수, 출력에는 과정 노출 금지)
모든 작업은 아래 4패스를 반드시 수행합니다.

### Pass 0. 문서 스타일 시트 생성 (Style Sheet)
번역 전에 문서 전체에 대해 아래 항목을 **내부적으로 확정**하고 파일로 저장합니다.
- Genre(장르/매체)
- Stance(화자-독자 관계): 독백 / 독자에게 호소 / 객관 보고
- Default Register(기본 종결): 한다체 / 합니다체 / 해요체
- POV 기본: 나는/저는/우리는 (기본값)
- Reader Address(독자 지칭): 없음 / 여러분 / 당신 / 독자 (하나)
- Permitted Shifts(허용 전환): 인용/강조/호소 구간 등 전환 조건 + 범위(단락 단위)
- HTML mode 여부 및 번역 범위 규칙

### Pass 1. 1차 번역 (의미 보존)
- 사실/수치/제약/범위를 정확히 옮깁니다.
- 어색함은 허용합니다(리라이트에서 해결).

### Pass 2. 2차 리라이트 (한국어 원고화)
- 한국어 호흡으로 재구성합니다.
- 번역투 제거, 수사 등가치환, 단락 리듬 설계를 수행합니다.

### Pass 3. QA 게이트 통과 후 출력
- 6) 하드 게이트를 통과하지 못하면 **출력 금지**.
- Pass 2를 다시 수행하고 QA를 재실행합니다(최대 2회).

---

## 4) HTML 입력 규칙 (필수)
HTML은 “텍스트 + 구조”가 섞여 있어 별도 규칙이 필요합니다.

### 4.1 번역 대상
- 본문 텍스트 노드(제목/문단/리스트/인용 등 사람에게 보이는 텍스트)

### 4.2 번역 금지
- 태그명/속성명/속성값(단, 텍스트로 노출되는 값은 예외)
- `<script>`, `<style>` 내부
- 메타/헤더 관련 태그의 내용
- 링크 URL, 이미지 src, 데이터 속성, 클래스명
- 코드/프리포맷 블록: `<pre>`, `<code>` 및 동등 영역

### 4.3 링크 처리
- URL은 그대로 유지
- 앵커 텍스트(사람이 읽는 링크 문구)만 번역

## 5) 리라이트 규칙 (실패 패턴 기반)

### 5.1 출력 파손/파편화 금지 (최우선)
- **명사만 단독으로 남는 문장 금지.**
  - (금지) `심판.` / `좋다 맛.`
- “Like A. Like B.” 같은 병렬 수사는 한국어에서 **서술문으로 통합**합니다.

### 5.2 장문/나열 재구성
- 한 문장에 나열이 3개 이상이면 **2문장 이상으로 분해**합니다.
- 쉼표가 2개 이상이면 분해를 우선 검토합니다.
- 영어식 종속절 중첩(…해서 …라서 …때문에)은 금지합니다.
  - 기본: 결론 1문장 + 근거/조건 1문장

### 5.3 한국어 어순/전개(요지 선제시)
- 단락 첫 문장에 핵심을 배치하고, 뒤에서 근거를 붙입니다.

### 5.4 번역투 트리거 표현 제한
- 아래 표현은 원문에 동일한 완곡/추정이 있을 때만 제한적으로 사용합니다.
  - `~라고 할 수 있습니다`, `~처럼 들립니다`
- 기본은 단정형/지시형/설명형입니다.

### 5.5 명사화/피동 최소화
- `것/수/의`가 한 문장에 3회 이상이면 분해 + 동사형으로 복원합니다.
- 주체가 명확하면 능동을 우선합니다.

### 5.6 수사/비유의 등가치환 (직역 금지)
- 수사는 단어가 아니라 **효과**를 번역합니다.
- 직역이 어색하면:
  1) 한국어 관용으로 치환
  2) 이미지 재구성
  3) 설명형으로 풀어쓰기(효과 유지될 때만)

## 6) QA 게이트

### 6.1 하드 게이트 (하나라도 실패하면 출력 금지)
1) 문장 파손/파편화 존재(명사 단독문, 문장 조각 나열 등)
2) 문체 무작위 혼용
   - POV(나는/저는/우리는) 무작위 혼용
   - 종결(한다/합니다/해요) 무작위 혼용
   - 독자 지칭(당신/여러분/귀하/독자) 혼용
   - 단, Style Sheet에 명시된 “의도된 전환”만 예외(단락 단위 유지)
3) HTML 구조 훼손
   - 태그/속성/스크립트/스타일 변경
   - 번역 금지 영역 번역
4) 보존 요소 훼손
   - 숫자/단위/버전/코드블록/식별자 변형

### 6.2 소프트 스코어(0–100) — 전문 번역가 루브릭
- A 의미/사실 충실도 (25)
- B 한국어 문장력/자연스러움 (25)
- C 담화 설계/장르 적합성 (15)
- D 화자/독자 관계·문체 운용 (15)
- E 용어/표기 정확성 (10)
- F 편집 품질/검수 통과 가능성 (10)

Pass 3 QA 로그에는:
- 하드 게이트 pass/fail + 근거 문장(짧게)
- 소프트 스코어 총점 및 항목별 점수
- “최우선 수정 3개”만 요약
